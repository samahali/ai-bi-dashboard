# Development & Deployment Guide

## Prerequisites

- Docker & Docker Compose
- Node.js 20+ (only needed if running the frontend outside Docker)
- Python 3.11+ (only needed if running the backend outside Docker)

## Quick start (Docker — recommended)

```bash
cp .env.example .env
# edit .env — at minimum set SECRET_KEY and a real WATSONX_APIKEY or
# OPENAI_API_KEY, or queries will fail with AIServiceError
docker compose up --build
```

- Frontend: http://localhost:3000
- Backend: http://localhost:8000
- API docs: http://localhost:8000/docs (dev only, disabled when
  `APP_ENV=production`)

`docker compose up` starts 4 services: `postgres`, `chromadb`, `backend`,
`frontend` (the dev-mode, hot-reload build of each). The production-target
services (`backend-prod`, `frontend-prod`, `nginx`) only run under the
`production` Compose profile — see [Production deployment](#production-deployment)
below.

## Database — dev vs. Alembic

In development (`APP_ENV` unset or `development`), the backend calls
`Base.metadata.create_all()` on startup — every model change shows up in
the DB automatically on next restart, no migration needed. This is
intentional for fast iteration and is gated off in production.

When you **do** change a model in `app/db/models.py`, generate a real
migration so production (which never runs `create_all()`) picks it up:

```bash
docker compose run --rm backend alembic revision --autogenerate -m "add x column"
docker compose run --rm backend alembic upgrade head   # apply locally too, for parity
```

Review the autogenerated migration before committing — Alembic's diffing
isn't perfect (it won't detect some column-type changes, check constraints,
or renames without help). Check it against `app/db/models.py` by hand.

To check current migration state:

```bash
docker compose run --rm backend alembic current
docker compose run --rm backend alembic history
```

```bash
# Backend
docker compose run --rm backend pytest

# Frontend
cd frontend && npm run typecheck && npm run lint
```

## Code style

- Backend: `black` (formatting), `ruff` (linting), `mypy` (type checking) —
  all in `requirements-dev.txt`.
- Frontend: `npm run lint` (ESLint), `npm run typecheck` (`tsc --noEmit`).

## Working without Docker (backend)

```bash
cd backend
python3.11 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt -r requirements-dev.txt
# Point DATABASE_URL / CHROMADB_HOST at reachable instances (e.g. via
# `docker compose up -d postgres chromadb` and localhost + the mapped ports)
uvicorn app.main:app --reload
```

Note: the app requires Python 3.11+ (uses `list[str]`-style builtin
generics throughout) — Python 3.8/3.9 will fail on import with
`TypeError: 'type' object is not subscriptable`.

## Working without Docker (frontend)

```bash
cd frontend
npm install
npm run dev
```

Set `VITE_API_BASE_URL` in a local `.env` if the backend isn't at the
default `http://localhost:8000/api/v1`.

## Adding a new LLM provider

`BIAgent` (`app/ai/agent.py`) drives SQL generation through one LangChain
`Runnable` chain regardless of provider. To add a provider:

1. Add an `_init_<provider>()` method returning a LangChain-compatible
   `LLM` or `BaseChatModel` (see `app/ai/langchain_llms.py` for the pattern
   used to wrap `ibm-watsonx-ai`, which has no official LangChain
   integration package pinned in this project).
2. Wire it into `_init_llm()`'s provider dispatch.
3. No changes needed to `_generate_sql` — the chain is provider-agnostic.

## Adding a new file type for upload

1. Add the extension to `ALLOWED_FILE_TYPES` in `.env` / `config.py`.
2. Add a branch in `FileParser._read_named_frames()`
   (`app/utils/file_parser.py`) using the appropriate pandas reader.
3. Add the extension to the frontend's upload dropzone accept list
   (`frontend/src/pages/datasets/UploadPage.tsx`).

## Production deployment

`docker compose --profile production up` brings up the production-target
services alongside the shared `postgres`/`chromadb`:

- **`backend-prod`** — built from `backend/Dockerfile`'s `production`
  stage: installs only `requirements.txt` (no dev/test tooling), runs as
  non-root `appuser`, and its entrypoint runs `alembic upgrade head` before
  starting uvicorn — schema migrations apply automatically on container
  start, no manual migration step in a deploy pipeline. Serves via
  `uvicorn ... --workers 2` (tune worker count to available CPU — not
  auto-scaled).
- **`frontend-prod`** — built from `frontend/Dockerfile`'s `production`
  stage: a static Vite build served by its own Nginx image.
- **`nginx`** — a top-level reverse proxy (`nginx/nginx.conf`) that fronts
  both: `location /` proxies to `frontend-prod:80`, `location /api/`
  proxies to `backend-prod:8000`, so the browser sees one origin.

`APP_ENV=production` matters beyond labeling: it disables `/docs`/`/redoc`
and skips the dev-only `create_all()` table creation in favor of the
Alembic migration the entrypoint just ran — see `app/main.py`.

### Building/running just the backend's production image

```bash
docker build --target production -t bi-backend:prod ./backend
docker run -d \
  -e DATABASE_URL=postgresql://user:pass@your-postgres-host:5432/bi_dashboard \
  -e APP_ENV=production \
  -e SECRET_KEY=<a real long random secret — do not use the dev default> \
  -e WATSONX_APIKEY=... -e WATSONX_URL=... -e WATSONX_PROJECT_ID=... \
  -e OPENAI_API_KEY=... \
  -e CHROMADB_HOST=your-chromadb-host -e CHROMADB_PORT=8000 \
  -p 8000:8000 \
  bi-backend:prod
```

### Environment variables that must change from `.env.example` defaults

| Variable | Why |
|---|---|
| `SECRET_KEY` | Defaults to a randomly generated value at process start if unset — **every JWT invalidates on every restart** unless this is set explicitly to a fixed, long, random value. |
| `DATABASE_URL` | Points at the dev `postgres` service hostname by default — must point at your real production Postgres. |
| `CHROMADB_HOST` / `CHROMADB_PORT` | Must point at a `chromadb/chroma:0.5.23`-compatible server (client is pinned to `0.6.3` for a reason — see [Security](SECURITY.md#dependency-version-notes)). Do not bump the ChromaDB server image without re-verifying client compatibility first. |
| `WATSONX_APIKEY` / `WATSONX_PROJECT_ID` / `OPENAI_API_KEY` | Placeholder values in `.env.example` — text-to-SQL queries fail with `AIServiceError` until at least one is a real, valid key. |
| `ALLOWED_ORIGINS` | Defaults to localhost dev ports — set to your actual frontend origin(s). |
| `APP_ENV` | Must be `production` to get the behaviors described above. |

### Database migrations in production

Handled automatically by `docker-entrypoint.sh` on container start (see
above). To run manually against a specific environment without starting
the app (e.g. from CI, before a blue/green swap):

```bash
docker run --rm \
  -e DATABASE_URL=postgresql://... \
  bi-backend:prod alembic upgrade head
```

Always back up the database before a migration that includes a
`downgrade()` you haven't tested — Alembic will run what's in the
migration file, not what you assume it does.

### Health checks

- Backend: `GET /api/v1/health` (used by `backend/Dockerfile`'s
  `HEALTHCHECK`).
- Frontend production image: `wget --spider http://localhost/` against the
  Nginx-served static build.

### Storage

`storage/uploads` and `storage/reports` are Docker named volumes
(`uploads_data`, `reports_data`) — persisted across container recreation,
but not automatically backed up. For a real production deployment, back
these up the same way you'd back up the database, or move to object
storage (S3-compatible) if running multiple backend replicas, since
local-disk storage doesn't share across replicas.
